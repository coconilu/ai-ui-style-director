# 工作流程

Web Style Director 在网站实现前增加一道风格选择门禁：

1. 理解用户要新建网站还是重构网站。
2. 只追问缺失的必要信息。
3. 推荐 5 个相关 UI 方向，并展示无品牌 SVG 草图、离线 HTML 画廊与上游 Light/Dark 实时预览。
4. 让用户选择一个方向，或要求再换一批未展示过的方案。
5. 生成项目专属的 `DESIGN.md` 与首屏草图。
6. 等待用户确认草图方向。
7. 根据确认后的契约和草图实现并验证 UI。

核心规则很简单：用户尚未选定方向、项目尚未生成 `DESIGN.md` 和首屏草图、草图方向尚未确认时，不开始实现 UI。

## 新建网站

有效的 brief 通常需要说明：

- 网站类型；
- 目标受众；
- 主要转化或工作流目标；
- 期望的密度或调性；
- 有限制时的技术栈和项目路径。

如果用户已经提供足够信息，应直接推荐，不要额外增加问卷。

## 重构网站

推荐重构方向前，至少检查一种真实来源：

- URL；
- 截图；
- 本地项目；
- 现有品牌资产；
- 必须保留的约束。

## 推荐与换一批

每个推荐展示方向名称、SVG 草图、主要 Light/Dark 实时预览、适配原因、首屏形态、可用组件库和主要风险，不暴露原始 catalog 数据。草图用于快速比较信息结构，实时预览用于查看更完整的色彩、字体与组件语言。纯终端客户端还会通过临时的本机回环 HTTP 链接拿到自包含 HTML 画廊，可在普通浏览器中一次比较全部 5 个方向。agent 会在用户选择期间保持前台服务运行，并在选定后或换一批前停止它。

如果用户不满意，skill 会排除当前 session 已展示的方向并推荐另一批。如果未展示的 catalog 选项已经耗尽，它会明确告知用户。

## 项目契约

用户选定后，`apply` 会写入：

```text
DESIGN.md
.ui-style-director/
  first-viewport-draft.svg
  selected-style.json
  recommended-components.json
  source-attribution.json
```

`DESIGN.md` 记录选定方向、真实视觉参考、首屏结构、布局、色彩角色、字体、组件建议和品牌安全约束。agent 在实现前展示 `first-viewport-draft.svg` 并等待确认，再在验证时记录有意偏离之处。

## 视觉预览

- 本地 SVG 卡片由仓库根据标准化风格元数据生成，可离线使用，不包含上游 logo 或截图。
- 上游 Light/Dark 链接只用于比较设计语言，不能作为最终网站资产。
- 每个风格保留 3 条真实参考，以避免把一个内部抽象方向伪装成某个品牌的精确克隆。

## 风格与组件的关系

Web Style Director 是风格路由器，不是另一个组件库：

- **Style profiles** 描述一致、适合 agent 阅读的视觉方向。
- **Component kits** 提供可能支持这些方向的实现材料。

选定风格负责控制页面方向；组件库只是可选的实现输入，应适配目标技术栈，而不能反过来引入新的视觉方向。
