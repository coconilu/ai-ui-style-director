# 工作流程

Web Style Director 在网站实现前增加一道风格选择门禁：

1. 理解用户要新建网站还是重构网站。
2. 只追问缺失的必要信息。
3. 推荐 5 个相关 UI 方向。
4. 让用户选择一个方向，或要求再换一批未展示过的方案。
5. 生成项目专属的 `DESIGN.md`。
6. 根据这份契约实现并验证 UI。

核心规则很简单：用户尚未选定方向、项目尚未生成 `DESIGN.md` 时，不开始实现 UI。

## 新建网站

有效的 brief 通常需要说明：

- 网站类型；
- 目标受众；
- 主要转化或工作流目标；
- 期望的密度或调性；
- 有限制时的技术栈和项目路径。

如果用户已经提供足够信息，应直接推荐，不要额外增加问卷。

## 重构网站

推荐重构方向前，至少检查一种真实来源：

- URL；
- 截图；
- 本地项目；
- 现有品牌资产；
- 必须保留的约束。

## 推荐与换一批

每个推荐只展示用户做选择所需的信息：方向名称、适配原因、首屏形态、可用组件库和主要风险，不暴露原始 catalog 数据。

如果用户不满意，skill 会排除当前 session 已展示的方向并推荐另一批。如果未展示的 catalog 选项已经耗尽，它会明确告知用户。

## 项目契约

用户选定后，`apply` 会写入：

```text
DESIGN.md
.ui-style-director/
  selected-style.json
  recommended-components.json
  source-attribution.json
```

`DESIGN.md` 记录选定方向、首屏结构、布局、色彩角色、字体、组件建议和品牌安全约束。agent 在实现前立即读取它，并在验证时记录有意偏离之处。

## 风格与组件的关系

Web Style Director 是风格路由器，不是另一个组件库：

- **Style profiles** 描述一致、适合 agent 阅读的视觉方向。
- **Component kits** 提供可能支持这些方向的实现材料。

选定风格负责控制页面方向；组件库只是可选的实现输入，应适配目标技术栈，而不能反过来引入新的视觉方向。
